# 서비스 정의서

이 문서는 `Chungmaru / 청마루`의 서비스 관점 기준 문서입니다.
발표 자료, README, 기능 범위 설명, PR 반영 판단의 기준으로 사용합니다.
세부 논문, 수치 실험, 참고 문헌은 이후 보강하고 여기서는 큰 틀과 설명 문장을 우선 관리합니다.

## 1. Service Overview

### 서비스 이름

Chungmaru / 청마루

### 서비스 한줄 설명

모바일 앱과 브라우저 익스텐션에서 수집한 온라인 텍스트를 문맥, 신뢰도, 플랫폼 제약까지 함께 고려해 경고, 마스킹, 허용, 보류로 중재하는 문맥형 유해 텍스트 중재 에이전트입니다.

## 2. Problem Definition

### 해결하려는 문제

온라인 댓글과 검색 결과 환경에서는 욕설, 혐오, 공격적 표현, 스팸성 문구가 빠르게 노출되며 사용자는 이를 읽기 전까지 위험 여부를 알기 어렵습니다.
하지만 유해 표현은 단어 하나만으로 결정되지 않습니다.
같은 단어라도 직접 공격, 사전 설명, 고유명사, 뉴스 인용, 검색 추천어, 댓글 조롱처럼 문맥에 따라 의미가 달라집니다.
또한 Chrome DOM, Android 접근성 트리, 동적 Google 검색 영역처럼 플랫폼마다 텍스트 수집 방식과 UI 반영 방식이 달라 단순한 키워드 차단이나 모델 호출만으로는 실사용 수준의 중재가 어렵습니다.

### 왜 중요한 문제인가?

유해 텍스트는 사용자 피로도와 불쾌감을 높이고, 서비스 사용 지속성에도 직접적인 악영향을 줍니다.
반대로 정상 문맥을 과하게 가리면 정보 접근성과 신뢰성이 떨어집니다.
따라서 실사용 가능한 필터링은 “욕설을 찾는 기능”이 아니라 수집, 정규화, 문맥 판정, 불확실성 처리, evidence span, 사용자 정책, UI 반영이 연결된 중재 파이프라인으로 설계되어야 합니다.

## 3. Target Users

### 주요 사용자 그룹

- YouTube, Instagram, TikTok 등에서 댓글을 자주 확인하는 일반 사용자
- 유해 표현 노출을 줄이고 싶은 민감 사용자 또는 보호 목적 사용자

### 사용자의 pain point

- 댓글을 읽기 전까지 어떤 표현이 유해한지 미리 알기 어렵다.
- 플랫폼마다 차단 방식이 달라 일관된 보호 경험을 제공받기 어렵다.
- 단순 금칙어 기반 필터는 정상 문장까지 가리거나, 우회 표현을 놓치기 쉽다.
- 모델이 애매하게 판단한 경우에도 왜 가려졌는지, 왜 허용됐는지 알기 어렵다.

## 4. Usage Scenario

### Scenario 1. 모바일 앱 기반 댓글 수집 및 분석

1. 사용자가 Android 앱에서 업로드 서버 주소를 설정하고 접근성 권한을 활성화합니다.
2. 사용자가 YouTube, Instagram, TikTok의 댓글 화면을 엽니다.
3. 앱이 화면의 댓글 텍스트를 수집하고 backend 판단 파이프라인으로 전달합니다.
4. 분류 결과, evidence span, 불확실성 사례를 기록해 플랫폼별 수집 품질과 모델 판단 품질을 비교합니다.

### Scenario 2. 브라우저 익스텐션 기반 실시간 마스킹

1. 사용자가 브라우저에서 텍스트가 많은 페이지나 댓글 영역을 엽니다.
2. Chrome Extension이 DOM 텍스트를 수집합니다.
3. 정규화, 정책, 모델 분류, evidence span 결과를 바탕으로 경고 또는 마스킹 UI를 적용합니다.
4. 사용자는 전체 차단이 아니라 설정 기반의 조절된 중재 결과를 확인합니다.

### Scenario 3. 애매한 문맥과 실패 사례 관리

1. 사용자는 정상 문맥이 가려지거나 유해 문맥이 누락된 사례를 발견합니다.
2. 해당 URL, 문장, 캡처, 기대 결과를 GitHub Issue 또는 Linear 코멘트에 남깁니다.
3. 사례는 safe/offensive/ambiguous regression set에 반영됩니다.
4. 이후 backend, extension, Android 수집 흐름의 어느 단계에서 실패했는지 추적하고 개선합니다.

## 7. Differentiation

### 우리 서비스만의 차별점

- 단순 마스킹이 아니라 `허용 / 경고 / 마스킹 / 보류`로 나누는 문맥형 중재 구조를 목표로 합니다.
- 같은 단어라도 직접 공격, 사전 설명, 고유명사, 인용, 검색 결과 문맥을 구분하는 safe-context와 regression set을 관리합니다.
- Chrome DOM과 Android 접근성 트리처럼 서로 다른 플랫폼 수집 환경을 함께 다루어, 실제 사용자 화면에서 발생하는 수집 실패와 UI 반영 제약을 검증합니다.
- backend의 boolean 판정뿐 아니라 evidence span과 판정 근거를 UI 적용 기준으로 삼아 전체 요소를 무작정 가리는 문제를 줄입니다.
- 모델 비교를 단독 성능 경쟁이 아니라 keyword-only, ML-only, full pipeline이 어떤 실패 유형에서 다른지 분석하는 실험으로 사용합니다.

- 대규모 운영용 관리자 콘솔과 신고 처리 시스템

## 9. Platform Decision

### 왜 이 플랫폼을 선택했는가?

Android 앱은 앱 외부의 실제 댓글 화면에서 텍스트를 수집하는 실험에 적합하고, Chrome Extension은 웹 환경에서 결과를 직접 시각적으로 반영하기 좋습니다.
또한 백엔드 API와 공통 계약 레이어를 분리하면 추후 분류 방식 변경이나 모델 교체 시에도 구조를 유지하기 쉽습니다.
따라서 청마루의 핵심 실험 대상은 모델 하나가 아니라 플랫폼 수집, 문맥 판정, evidence span, UI 중재가 연결된 전체 파이프라인입니다.

## 13. PR 반영 및 문서 동기화 원칙

PR은 단순 머지 대상이 아니라 서비스 범위를 바꾸는 입력으로 취급합니다.
리뷰 시 아래 기준으로 반영 여부를 판단합니다.

### 반영

- 현재 서비스 목표와 직접 연결되는 기능
- 기존 모듈 경계를 크게 깨지 않는 변경
- Demo 또는 핵심 사용자 시나리오를 강화하는 변경

### 보류

- 현재 범위를 벗어나는 기능 확장
- 다른 담당 영역과 계약 충돌이 큰 변경
- 테스트 또는 영향 범위 설명이 부족한 변경

### 문서 업데이트 규칙

- PR이 서비스 목적, 대상 사용자, 시나리오, 데모 범위를 바꾸면 이 문서를 먼저 갱신합니다.
- README는 항상 이 문서로 연결되는 진입점 역할만 유지하고, 긴 설명은 여기서 관리합니다.
- 발표 자료나 중간보고용 텍스트는 이 문서 문장을 우선 재사용합니다.
